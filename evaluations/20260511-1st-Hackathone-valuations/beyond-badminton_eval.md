# 評価レポート: beyond-badminton

**評価日**: 2026-05-11
**実施回数**: 2回

---

## 概要

20 体の LLM エージェントを 14×14 の 2D 仮想空間に配置し、バドミントンコートまたは名前を伏せた月面的空間を舞台として、AI 集団が独自のスポーツや文化をどこまで自律的に立ち上げられるかを観察する実験プロジェクト。エージェントは gemma4:e4b で駆動され、通信半径 4 の同一 place 内でのみメッセージ交換が可能、capacity 6 の矩形コートや PA 放送（step 34/67）といった環境ルールが課される。主要メカニズムは「ペルソナの有無 × 既知バドミントン語彙の有無」の 2×2 比較実験（Patterns 6-9）で、各セルで 100 ステップの相互作用を実行し messages.jsonl のキーワード集計から立ち上がる文化を比較する。観察された現象には、応援文化（cheer/hype）、新動作 "Punctuation Pass" の命名と 16 体への伝播、構造化探索（rod/structure/mesh）、エネルギー連鎖の集団演技シーケンス共同設計が含まれる。テーマは「シンギュラリティ後のスポーツ創発」であり、ペルソナと文脈の組合せが立ち上がる文化の対象と形式を規定する条件を探る。

## 評価詳細

### A. 創発設計 — 最終: 6.5/10

#### 評価
世界ルールの設計精度は高い。14×14の有界フィールド、`communication_radius`、矩形 place の `half_size_x/y`、`capacity` 強制、同一 place 内でのみ通信可能というルール、そして place 内エージェントにのみ占有率・容量を生データとして提示する設計（`agent.py:256-270`）は、エージェントが意味のある判断を下せる十分な制約構造を提供している。一方で行動の自由度は中程度に留まる。プロンプトに「Inaction is boring」「Avoid abstract discussions about 'the meaning of place'」「ACT INSTEAD」といった行動指示が明示的に埋め込まれており（`agent.py:312-327, 451-469`）、純粋な「データのみ提示」という理想からは外れる。ただし 2×2 比較実験デザイン自体は創発条件の探究として優れており、ペルソナ有無 × 文脈有無の組合せで異なる文化が立ち上がる構造的余白は確保されている。

#### 根拠
- `simulation/agent.py:265-270` — place 内エージェントには `occupancy_rate: {x:.2f}` などの定量データのみが渡される。place 外には place 情報を渡さない設計で、知覚境界が明示されている。
- `simulation/agent.py:312-327` — 「ACTION IMPERATIVE」セクションで具体的な行動候補（"Walk into..."、"Pick up, hold..."、"Approach another agent and propose an activity"）を列挙し、「Inaction is boring」と明示的に行動を促している。
- `simulation/simulation.py:485-519` — 容量超過時の移動ブロック処理。世界ルールが厳格に強制される。
- `simulation/agent.py:99-122` — 通信ルール（同一 place 内 or 両者外部のみ通信可能）が明確に実装されている。
- `README.md` — "ペルソナと文脈の組合せで、立ち上がる文化の **対象と形式** が決まる" — 4セルの比較で創発の条件を探る研究デザイン。

### B. 世界設定 — 最終: 8.0/10

#### 評価
独自性とテーマ性が際立つ。「シンギュラリティ後の月面ハビタットで、人類の去った後に残された人工生命体が、名前を伏せられた競技用具（mesh-tipped long objects、small flying objects）に初めて遭遇する」というシナリオは、ハッカソン提出物としては非常に独創的。これに通常のバドミントン施設という対照条件を組み合わせた 2×2 比較実験は、社会科学・組織論・文化形成の観点から「集団に文化はどう立ち上がるか」を問う研究的価値が明確。`Punctuation Pass` の命名→16体採用という観察は文化的創発の事例として説得力を持つ。実験条件として「100ステップ ≈ 6〜14時間」という重い計算コストを払って 4 セルを実走しているのも評価が高い。

#### 根拠
- `simulation/config.yaml:17-28` — Pattern 7 の `world_description`：「Lunar Habitat Beta — a sealed dome on the surface of the Moon. Long after humanity left... You and your peers are the first to encounter this space.」
- `README.md` — "Pattern 7（名前を伏せた空間 × ペルソナあり）— 動きの語彙形成 / Agent 19（Casual ペルソナ）が **\"Punctuation Pass\" を命名**（step 65）/ 翌ステップで **16 体が一斉採用**"
- `README.md` — "ペルソナ有では応援や動きの語彙形成を伴う関係的文化、ペルソナ無では構造化を中心とする文化（空間構造 / 動作シーケンス）が観察される"（社会科学的観察結果）
- `simulation/config_pattern8.yaml:11-20` — P8 を P7 の "ablation" と明示し、2×2 ファクトリアル設計を意図的に組んでいる。

### C. 発展性 — 最終: 6.5/10

#### コード拡張性
モジュール分割は明快。`agent.py`（エージェント論理）、`simulation.py`（シミュレーションループ）、`ollama_client.py`(LLM 通信)、`visualization.py`、`utils.py` が分離されている。設定の外部化も徹底しており、`places`、`personas`、`announcements`、`world_description` 等が全て YAML で記述可能。`world_description_override` により新シナリオを差し込める設計（`agent.py:276-289`）。複数 place、複数 fire、矩形 place（`half_size_x/y`）への拡張対応も済んでいる。新規 place type やシナリオを追加するコストは低い。

#### 将来展望
README に「**複数 seed / 複数 run での再現性検証は今後の課題**」と明記（README.md L184-185）されており、限界の認識は明確。しかし将来展望のセクション自体は限定的で、メモリ機構の改良、エージェント学習機能、より長期のシナリオなど、研究を深める具体的な技術的拡張プランの記述が乏しい。

#### 根拠
- `simulation/agent.py:276-289` — `world_description_override` により設定で世界記述を完全に差し替え可能。
- `simulation/config_pattern6.yaml` / `config_pattern8.yaml` / `config_pattern9.yaml` — 同一コードベース上で 4 つのシナリオを設定差分のみで実行している。
- `simulation/utils.py:34-39` — 矩形 place への拡張（`get_place_half_sizes`）で後方互換性を保ちつつ拡張。
- `README.md` — "複数 seed / 複数 run での再現性検証は今後の課題" — 限界の認識はあるが、具体的な発展方向の記述はやや控えめ。

### D. 技術実装 — 最終: 7.5/10

#### LLM利用
プロンプト設計は丁寧で、`message_prompt`（位置情報なし）と `decision_prompt`（位置情報あり）を 2 段階に分離してフェーズ間の情報リークを防いでいる（`agent.py:234-491`）。JSON 出力はネストされた波括弧と文字列を正しく扱うブレース・マッチング・パーサで抽出（`agent.py:493-528`）。フォールバック解析もあり、JSON 失敗時にも `direction` キーワード抽出を行う（`agent.py:530-606`）。Ollama に `think:false` を渡して CoT トークン消費を回避し、`num_ctx`/`num_gpu` も明示的にチューニング（`ollama_client.py:79-89`）。`temperature 0.2` を採用し再現性寄りの設定。

#### メモリ設計
メモリは平坦な文字列リスト（`self.memory: List[str]`）で、`memory_limit=20` までストアし、プロンプトには直近 `memory_size=5` のみを使う構成（`agent.py:159-165`）。`memory` はエージェント自身が次ステップに残したい内容を LLM 自身に書かせる「自己フィードバック」型（`agent.py:649-658`）。受信メッセージも分離して `received_messages`（`message_history_limit=10`）に保管。シンプルだが意味のある設計で、単なるログダンプではない。ただし階層的メモリやエピソード記憶のような高度な仕組みはなく、要約や反省ステップも入っていない。

#### コード品質・シミュレーション正確性
コードは整っており、`TypedDict` で意思決定構造を型付け、定数を冒頭にまとめ、各メソッドに docstring がある。シミュレーションループは「①全員メッセージ生成→②メッセージ送信→③全員アクション生成→④移動実行（容量チェック）」と同期的フェーズ分割（`simulation.py:371-519`）で、順序依存のレースコンディションを意識した設計。`simulation.py:485-489` の「Sequential processing means earlier moves consume capacity before later ones are checked」というコメントは正確性への配慮を示す。エラーハンドリングは LLM 呼び出しとエージェント決定の両方に存在（`ollama_client.py:102-107`, `agent.py:622-628, 661-663`）。再現性についてはシードが固定されておらず（README で明示）、これが減点要素。

#### 根拠
- `simulation/agent.py:493-528` — 文字列内エスケープを考慮したブレース・マッチング JSON 抽出。
- `simulation/ollama_client.py:79-89` — `think:false` と Ollama options の細やかなチューニング。
- `simulation/simulation.py:371-519` — 4 フェーズの同期実行（メッセージ決定 → 送信 → アクション決定 → 移動）。
- `simulation/agent.py:649-658` — LLM 自身に次ステップ用メモリを書かせる自己フィードバック型メモリ。
- `simulation/simulation.py:485-489` — 容量強制のシーケンシャル処理の意図的なコメント。
- `README.md` — "seed は固定していないため、LLM 出力には確率的な揺らぎがある" — 再現性の限界を自認。

## 総評

### 優れている点
- **2×2 比較実験デザインの研究的厳密性**：ペルソナ有無 × 文脈有無の 4 セル全てを 100 ステップ実走し、観察結果を定量キーワード集計で裏付けている。ハッカソン提出物として研究らしい設計が際立つ。
- **独創的な世界設定**：シンギュラリティ後の月面ハビタットで「名前のない用具に初めて遭遇する人工生命体」という設定は、文化形成・社会的命名・集団演技などの社会科学的問いと自然に接続する。
- **2 段階意思決定と知覚境界の明示**：メッセージ判断時に位置情報を渡さず、place 外エージェントには place 内データを渡さない構造は、創発を見るための情報設計として理にかなっている。
- **再現環境の整備**：YAML 設定の外部化、複数パターン用 config の整理、Ollama クライアントの GPU/コンテキストチューニングなど、実走可能性が高い。

### 改善点・提言
- **プロンプト中の行動指示が強い**：「Inaction is boring」「Avoid abstract discussions about 'the meaning of place'」「ACT INSTEAD」などの直接的な指示は、純粋な創発設計の観点では行動の自由度を狭めている。同等の効果を「世界の物理的圧力（時間制限、評価される条件など）」で実現できれば、創発設計のスコアは一段上がる。
- **再現性検証**：seed 固定と複数 run の集約は README でも今後の課題として挙げられているが、ハッカソンとしても 2-3 run 集約があれば観察の頑健性が大きく上がる。
- **メモリ機構の発展**：現在は直近 5 件の平坦な文字列リスト。エージェント自身の命名・採用・命名理由を時系列で保持・要約する仕組みがあれば、文化形成の長期的観察に寄与する。
- **将来展望の具体化**：README に「次にどのような研究的拡張を行うか」（例：別モデルでの再現、命名語彙のネットワーク分析、エージェント間の発信影響度測定など）を具体的に書くと発展性スコアが伸びる。

### 一言コメント
ハッカソン提出物として、研究の問いの立て方・実験デザイン・観察結果の三拍子が揃った骨太な作品。創発条件の探究という難題に正面から取り組んでおり、プロンプトの行動指示の強さと再現性検証の不足を改善すれば、研究プロジェクトとしてさらに伸びる余地がある。
