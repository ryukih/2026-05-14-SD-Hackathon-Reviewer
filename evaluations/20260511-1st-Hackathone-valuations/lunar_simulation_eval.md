# 評価レポート: lunar_simulation

**評価日**: 2026-05-11
**実施回数**: 2回

---

## 概要

月面の 30×30 2D グリッド上で、定員 10 人の中央基地 `central_lunar_base` を共有する 20 人の LLM エージェントによる酸素制約シミュレーション。各エージェントは初期酸素 100 を保持し、基地外では 1 step あたり 2.5 の酸素を消費し、基地内（定員内）では消費が止まる構造になっている。エージェントは半径 5 の範囲で会話可能であり、基地入口付近では半径 2 で内外通信ができ、移動・会話・記憶を通じて行動を選択する。LLM プロンプトは数値データ（占有率、酸素残量、枯渇までの推定 step 数）のみを提示し、戦略的指示を含めず行動を LLM 判断に委ねる設計となっている。デフォルトでは Gemini Developer API の `gemini-3.1-flash-lite-preview` を使用し、8 並列 worker で 100 step を実行する。出力にはフレーム画像、メッセージログ、記憶・推論ログ、統計グラフ、再現用設定が含まれ、背景画像合成や 20 回実験のグリッド動画生成にも対応している。テーマは、有限資源（酸素・基地定員）下における自律エージェント群の協調・競合行動の創発である。

## 評価詳細

### A. 創発設計 — 最終: 8.0/10

#### 評価
世界ルールは非常に精密に設計されている。正方基地と入口ゾーン (`utils.py:78-106`)、収容定員、酸素消費レート3層 (`config.yaml:15-20`)、エリア別の会話可能範囲（基地内同士／基地外同士／入口越境）が明確に分離されている。プロンプトはエージェントに数値データのみ (`occupancy_rate`、`oxygen`、`~N steps until depletion`) を提示し、「どう振る舞うべきか」の指示を排除しているため、行動戦略は LLM の判断に委ねられている (`agent.py:340-348`, `agent.py:443-451`)。20人 vs 定員10人という構造的ジレンマが設定上の創発圧力を生むため、設計レベルでの創発ポテンシャルは高い。一方、`collective_prompt` フラグや `~N steps until depletion` のような前計算はわずかながら誘導性を持ち、純粋な raw data 提供から少しだけ後退している。

#### 根拠
- `agent.py:340-348` — メッセージ判断タスクは "Decide what message to send..." と最小限の指示のみで、戦略の指定なし
- `agent.py:443-451` — アクション判断は `stay | move: up/down/left/right` と JSON 仕様のみ。「定員超過なら退去せよ」等の指示はない
- `agent.py:236-254` — 酸素状態は数値とともに `steps_to_death` が前計算されエージェントへ渡される（軽い行動誘導）
- `agent.py:325-326` — `collective_prompt=true` 時のみ "operating as part of a group under survival constraints" を付与（オプショナル）
- `simulation.py:497-506` — 酸素消費は環境ルールとして自動適用、エージェントが酸素を「使う」のではなく世界が課す制約
- `config.yaml:5-13` — 通信半径・入口越境半径などの世界パラメータがすべて外部定義

### B. 世界設定 — 最終: 7.0/10

#### 評価
月面酸素制約という設定は標準デモにはほぼ見られない題材で、20人エージェントに対し中央基地定員10人という構造は「シェルター不足下の資源配分」の社会科学的アナロジーとして機能する（災害避難所、ICUベッド配分、難民キャンプ等）。テーマは明確で緊張感もある。ただし世界の物理は比較的単純（1基地のみ、地形差なし、4方向移動のみ）で、月面という設定自体は意匠的で、火災・嵐シナリオなどに置換可能な抽象度。原型は良いが、世界の深さ（地形、複数施設の連携、社会的役割の差異）はまだ浅い。

#### 根拠
- `README.md` — "月面 2D 空間で、20 人の LLM エージェントが移動・会話・記憶をしながら、定員 10 人の中央基地をめぐって行動する酸素制約シミュレーションです。"
- `config.yaml:21-27` — 単一の `central_lunar_base` のみ、capacity 10
- `config.yaml:6` — `num_agents: 20`（定員の倍）という鋭い構造的緊張
- `agent.py:19-24` — 4方向移動のみで地形要素なし
- `events/rescue_announcement.py` — Mission Control による救助メッセージという実験介入要素があり、社会的アナウンスの効果検証も可能な設計

### C. 発展性 — 最終: 7.0/10

#### コード拡張性
モジュール分割が明瞭で、`agent.py`、`simulation.py`、`llm_client.py`、`events/`、`utils.py`、`visualization.py` が責務分離されている。`places` はリスト型で複数施設を許容 (`simulation.py:54-78`)、新しい施設タイプを YAML 追記のみで追加可能。イベントは登録レジストリパターン (`events/registry.py:10-12`) で、`SimulationEvent` 基底クラスを継承して新規イベントを追加し `EVENT_TYPES` に登録するだけで拡張できる。LLM プロバイダも Ollama / LM Studio / Gemini / Mock の4種をサポートし、追加も容易。設定ファイル `config.yaml` / `config_collective.yaml` / `config_low_thinking.yaml` の3バリアントが既に提供されている。

#### 将来展望
README はイベント追加方法 (`event_catalog.yaml` 参照) のみに触れ、明示的な「将来展望」セクションがない。ビジネス応用や研究的拡張の言及がなく、ロードマップが薄い。20回比較レポート (`make_20run_comparison_report.py`) や複数の集計レポート生成スクリプトは存在し、実験フレームワークとしての完成度は高いが、それを README で訴求できていない。

#### 根拠
- `events/registry.py:10-37` — `EVENT_TYPES` 辞書登録パターンによる宣言的拡張
- `events/base.py:13-55` — `SimulationEvent` 基底クラスで `should_fire` / `apply` / `manifest` を分離
- `simulation.py:54-78` — `places` はリストで複数基地に対応
- `llm_client.py:748-803` — 4プロバイダ対応 + フォールバック
- `event_catalog.yaml:1-43` — イベント拡張カタログが整備
- `README.md` — 明示的な「Future Work」「想定応用」セクションが無い
- `config_collective.yaml`, `config_low_thinking.yaml` — 実験バリアント設定が既に整備されているが README では深く語られていない

### D. 技術実装 — 最終: 8.0/10

#### LLM利用
2段階プロンプト設計が秀逸：Phase 1 のメッセージ判断ではエージェント自身の座標を渡さず (`agent.py:280, 332`)、Phase 3 のアクション判断で初めて座標を含める (`agent.py:361, 416`)。これにより「通信ではプライバシー的に位置を伏せ、行動判断では位置を使う」という設計意図が表現されている。Gemini API の `responseMimeType: application/json` と `thinkingLevel` を活用 (`llm_client.py:357-366`)、HTTP エラーへの指数バックオフ・リトライ (`llm_client.py:378-433`)、並列ワーカー (`simulation.py:291-306`) も整備。`MockLLMClient` (`llm_client.py:482-746`) によりオフライン検証も可能。

#### メモリ設計
メモリは2系統：(1) LLM が自ら生成する `memory` フィールド（自己メモ、`agent.py:621-630`）、(2) 受信メッセージキュー (`agent.py:650-666`)。両者とも FIFO で上限制御され、プロンプトには直近 N 件のみ展開される (`agent.py:215-216`, `agent.py:223-227`)。LLM 出力に基づく自己フィードバック型メモリは単純なログダンプより意味のある設計。

#### コード品質・シミュレーション正確性
`TypedDict` による型定義 (`agent.py:27-39`, `utils.py:7-25`)、命名は明瞭、責務分離が良い。JSON パースで波括弧マッチングを実装する堅牢性 (`agent.py:454-489`)、フォールバック値も整備。シミュレーションは4フェーズ（メッセージ判断→送信→アクション判断→移動→酸素消費）を同期的に実行 (`simulation.py:572-738`) し、競合状態が発生しない設計。定員超過時の入場ブロック (`simulation.py:523-570`) も明示的。一方、`random.seed` などの再現性制御が無いため、同じ config でも実行毎に結果が変わる（複数 run 比較スクリプトの存在を踏まえると意図的かもしれないが、再現性の観点ではマイナス）。シークレットの redact (`main.py:102-114`) や設定保存 (`main.py:117-178`) は実験記録の質を高めている。

#### 根拠
- `agent.py:270-348` — メッセージプロンプトに座標を含めない実装
- `agent.py:350-452` — アクションプロンプトでは座標と基地位置を含む
- `llm_client.py:357-366` — Gemini の構造化出力・思考レベル設定
- `llm_client.py:378-433` — リトライ・指数バックオフ
- `simulation.py:291-306` — `ThreadPoolExecutor` による LLM 並列実行
- `agent.py:454-489` — ネスト対応の JSON 抽出
- `agent.py:621-630` — LLM 出力の `memory` を次ステップに引き継ぐ自己フィードバック型メモリ
- `simulation.py:572-738` — 同期4フェーズの step 実装
- `simulation.py:523-570` — 定員チェック付き移動
- `main.py:102-114` — API key 等の redact 処理

## 総評

### 優れている点
- 世界ルール（酸素3層、定員、入口ゾーン、通信境界）が極めて精密に設計され、エージェントには数値データのみが渡されるため、設計レベルでの創発ポテンシャルが高い
- LLM 統合が成熟しており、2段階プロンプト（位置秘匿の通信→位置あり行動判断）、構造化 JSON 出力、リトライ、並列実行、Mock フォールバックなど実運用品質
- モジュール分割とイベントレジストリパターンにより、新規イベント・施設・LLM プロバイダの追加が容易
- 20回比較レポートや複数 config 変種（collective_prompt / low_thinking）が既に整備されており、実験フレームワークとしての完成度が高い
- `TypedDict`、シークレット redact、設定再現用ファイル出力 (`config_used.yaml`, `experiment_conditions.md`) など実験再現性への配慮

### 改善点・提言
- README に明示的な「Future Work」「想定応用領域（資源配分・避難所運用・ICUベッド管理など）」セクションを追加し、研究・ビジネス価値の訴求を強化すべき
- 乱数シード制御 (`random.seed`) が無いため、同 config でも完全再現できない。シード config 化を推奨
- 世界の物理が比較的単純（1基地・地形なし・4方向移動）。地形コスト、複数施設の役割分化、エージェント個性などを追加すれば世界設定スコアの底上げが可能
- `~N steps until depletion` のような前計算情報は便利だが軽い行動誘導でもある。raw な `oxygen_level` と `consumption_rate` のみを渡す版を実験オプションとして用意すれば創発潜在性がさらに高まる
- `collective_prompt` の効果検証結果が README に記載されておらず、実験設計の意図が読者に伝わりにくい

### 一言コメント
酸素制約と定員不足という鋭い構造的ジレンマを、精密な世界ルールと最小限の行動指示で構築した、設計思想が明快な良作。LLM 統合と拡張性も実運用レベルで、README で将来展望と実験結果の解釈をもう一歩深めれば、さらに高い評価を得られる完成度。
