## 概要

2 次元グリッド空間上に複数の場所（デフォルトでは左右に 1 つずつのバー、各々独立した収容上限を持つ）を配置し、LLM エージェント群が活動するマルチエージェント・シミュレーション。シミュレーション途中で 1 つ以上の火事イベントが指定ステップで発生し、知覚半径内のエージェントには位置・強度・距離などの定量情報のみが渡される（「危険」等の定性的記述は与えられない）。各エージェントには初期化時に性別ペルソナ（male/female）がランダムに付与され、プロンプトの自己状態欄や近隣情報に反映される。エージェントは毎ステップ LLM により Message / Memory / Action を生成し、4 方向移動＋滞在の離散行動と、近傍エージェントへのメッセージ送信を同期的に実行する。通信は通信半径内かつ「両者とも場所外」または「両者とも同じ場所内」の条件下でのみ成立し、場所内エージェントのみが占有率等の数値情報を直接受け取る。Memory フィールドは次ステップの自己フィードバックとして利用され、共有 LLM エンジンの下でも履歴差分から個性が創発する設計となっている。本派生は日本語プロンプト切替、複数モデル（qwen2.5、gemma3、elyza、swallow、sarashina 等）でのスモーク／本走比較、ja×高密度（d50）3-run 観察を追加し、定量情報のみから生まれる集団行動の創発をテーマとする。

## 評価詳細

### A. 創発設計 — 最終: 9.0/10

#### 評価
世界ルールの設計精度・行動の自由度ともに高水準で、創発ポテンシャルは設計段階で最大級に確保されている。エージェントには「占有率」「火災強度」「距離」など**生の数値のみ**が与えられ、「危険」「避難すべき」といった**定性的な解釈や行動指示は完全に排除**されている。一方で行動空間（move/stay × 4方向、任意メッセージ、自由形式メモリ）は完全に開かれており、戦略はLLMに完全委任。火災知覚モデルB（距離依存）と、場所内/外＋同一場所限定の通信ルールが、情報の非対称性という強い創発トリガを構造的に作っている。提出者の `SUBMISSION.md` 観察「強度0.8をエージェントが自ら "manageable" と再解釈」「火災情報を直接知覚せず会話経由で知る伝播の連鎖」は、この設計が実際に creative reasoning を引き出すことを実証している。

#### 根拠
- `agent.py:50-56` — `fire_entry` テンプレが `Position / Intensity / Radius / Distance` のみ。定性的形容詞ゼロ。
- `agent.py:43-48` — `place_status_template` も `Number of agents here / Capacity / Occupancy rate` のみ。
- `agent.py:376-399` — `_build_fire_section` のコメントに「Only quantitative data is provided ... No qualitative descriptions」と明記。
- `simulation.py:299-322` — `get_fire_info_for_agent` は知覚半径外には `None` を返す（Model B）。
- `agent.py:286-309` — `get_nearby_agents`: 通信は「両方とも場所外」または「同じ場所内」のみで成立、異なる場所間や場所内外の通信は不可能というルールが明確。
- `README.md` — 「エージェントには生の数値データ（占有率、火災の強度・距離等）のみを提供し、行動指示や定性的評価（「危険」「快適」等）は一切与えない」
- `SUBMISSION.md` — Agent 0 が intensity 0.8 を "manageable" と自発的解釈する観察例（規範の逸脱の正常化の証拠）

### B. 世界設定 — 最終: 7.0/10

#### 評価
シナリオは「2Dフィールド上に複数のバーがあり、途中で火災が発生する」というもので、災害・群衆系のシミュレーションとして単体では特別独創的ではない。ただし、(1) 場所ごとに収容上限を持ち占有率という社会的圧が常時発生、(2) 火災が定量情報のみで知覚半径B方式で配信され、知覚した者・しなかった者で情報非対称が生まれる、(3) ペルソナ（gender）が会話に乗る、という3要素の組み合わせは「定量情報のみからの集団規範の創発」という社会科学的問いに直結している。`SUBMISSION_ja_d50_qwen25_7b.md` の観察では「右側バーの存在しない『舞踏場』『個室』を集団で共有」「『出口・消防設備・煙・温度』という防災語彙の集合的補完」など、情報拡散・共有現実構築の研究素材としての価値が確認された。テーマ自体の独創性は中庸だが、観察可能性の設計は丁寧。

#### 根拠
- `config.yaml:21-33` — left_bar (capacity 12) と right_bar (capacity 10) の非対称な収容上限。
- `config.yaml:48-61` — 火災2件、`fire_1` は right_bar 直上を覆う半径15、`fire_2` は反対側の小さな半径8。火災の位置・強度に意図ある設計。
- `README.md` — 「LLMエージェントの**創発性**と**自律性**を発現させることを目的とした」「フィールドに複数の場所が存在し、各場所は独立した収容上限を持つ」
- `SUBMISSION_ja_d50_qwen25_7b.md` — 「出口 634 / 消防 584 / 煙 477 / 温度 443」など、プロンプトに存在しない防災語彙が集団で創発した具体的観察。
- 限界: 場所のtype（bar/cafe/library）は文字列ラベルのみで、type固有の機能差は実装されていない。「複数typeへの拡張準備」に留まっており、世界の経済・社会動態のモデル化はまだ薄い。

### C. 発展性 — 最終: 8.0/10

#### コード拡張性
モジュール分離は良好。`agent.py`（LLMロジック）、`simulation.py`（世界ループ）、`ollama_client.py`（LLMバックエンド）、`visualization.py`（可視化）、`utils.py`（型と判定関数）が責務ごとに分かれている。`places` と `fires` がともにYAMLのリスト形式で、新しい場所や火災の追加は設定変更のみで可能。`PlaceConfig` / `FireConfig` を `TypedDict` で定義しているため、新typeの追加も型安全に行える。提出者は `agent.py` のみの侵襲で `PROMPT_TEMPLATES` 辞書ベースの言語切替を後付けしており、これは現コードベースの拡張容易性を実証する好例。多数の `config_*.yaml` バリアント（gemma3, qwen2.5/3, llama3.2, sarashina, swallow, elyza, phi4 など）がリポジトリに同梱されている点もモデル横断の拡張実績として高評価。

#### 将来展望
READMEに「将来的にカフェや図書館などにも拡張可能」と具体的な`type`案を明示。`PROMPT_LANGUAGE.md` には日本語化の設計判断とトレードオフが記録されており、別言語への横展開が容易。`CLAUDE.md` には「`communication_radius` を変えると伝播速度はどう変わるか」「`temperature` の影響」など今後の感度実験計画が記載されている。一方、ビジネス・社会科学へのマッピング（避難計画、商業施設の混雑緩和、SNS情報拡散など）への発展構想までは踏み込んでいない。

#### 根拠

### D. 技術実装 — 最終: 9.0/10

#### LLM利用
プロンプトエンジニアリングは非常に丁寧。**Phase 1（メッセージ決定、位置情報なし）と Phase 3（行動決定、位置情報あり）の2段プロンプト**に分割することで、メッセージ生成時に座標を見て「先回りした計算結果」を漏らすことを防いでいる（`simulation.py:324-422`）。LLM出力はJSON固定スキーマで指示し、`_extract_json_from_text` がネストブレース対応・文字列内ブレース無視・エスケープ対応の堅牢なパーサになっている（`agent.py:600-635`）。JSON失敗時はフォールバックパーサで方向キーワードを救う（`agent.py:637-651`、`682-713`）。温度0.2、`repeat_penalty` 1.1、`min_p` 0.05、`repeat_last_n` 128 などサンプリングパラメータが小型LLMの安定性に合わせて調整済み。API呼び出しは `try/except` でフォールバック返答（`agent.py:725-731`、`760-762`）。

#### メモリ設計
LLM自身が生成した `memory` フィールドを次ステップの `Previous Memory` として自己フィードバックする設計（`agent.py:748-757`）。`memory_limit`（保存上限）と `memory_size`（プロンプトに使う直近件数）が分離されており、長期保存と短期参照の両立が可能。同様に `message_history_limit` と `message_context_size` も独立。受信メッセージは送信元IDつきで保存され、`from Agent X: ...` の形式でプロンプトに整形（`agent.py:364-374`、`777-793`）。「単なるログダンプ」ではなく、エージェントが内省的に書き直した記憶が次ステップの推論を駆動する仕組みになっている。

#### コード品質・シミュレーション正確性
命名・型ヒント・docstringが一貫しており可読性が高い。`TypedDict`（`MessageDecision`/`ActionDecision`/`PlaceConfig`/`FireConfig`）で構造を明示。`DIRECTION_MAP`、`MAX_MESSAGE_WORDS`、`MAX_POSITION_ATTEMPTS` 等の定数化。シミュレーションは厳密に4フェーズの同期実行（メッセージ決定→送信→行動決定→移動）で、`simulation.py:324-422` のコメントに順序の意味が明記されている。火災発火チェック、エージェント状態更新の順番も明示的。`output/messages.jsonl` と `output/memory_reasoning.jsonl` が一次データとして体系的にロギングされており、観察の再現・検証が可能。**減点要因**: 乱数シードを明示的に固定する仕組みは見当たらず（`config.yaml` に `seed` キーなし）、再現実験には外部からの `random.seed` 注入が必要。テストコードは同梱されていない。

#### 根拠
- `agent.py:600-651` — JSON 抽出（ネストブレース対応）と方向キーワードフォールバック。
- `agent.py:415-490` / `492-598` — Phase 1（位置情報なし）と Phase 3（位置情報あり）の分離プロンプトビルダー。
- `simulation.py:324-422` — 4-Phase 同期実行の明示順序。
- `agent.py:733-762` — `decide_action` が `decision['memory']` を `Step N: {memory_content}` 形式で履歴に追加し、`memory_limit` で先頭削除。
- `ollama_client.py:45-97` — `repeat_penalty` / `repeat_last_n` / `min_p` を options に含め、`RequestException` を捕捉して空文字を返す堅牢な実装。
- `simulation.py:130-176` — `_log_message` と `_log_memory_reasoning_batch`（バッチI/Oで効率化）の構造化ロギング。
- `config.yaml:36-45` — LLM サンプリングパラメータの調整値。
- 減点: シードの欠如、テストの欠如。

## 総評

### 優れている点
- **創発設計の徹底**: 数値のみ提示・行動指示ゼロという原則が `agent.py` のテンプレ単位で守られている。実際に観察された「intensity 0.8 を 'manageable' と自己解釈」「存在しない舞踏場/個室/消防設備の集団的共有」など、設計意図通りの創発が複数モデル・複数runで再現されている。
- **2フェーズプロンプト分割**: メッセージ決定（位置情報なし）と行動決定（位置情報あり）を分けることで、エージェントの「会話で何を共有するか」をLLMの内部知識に依存させる工夫が秀逸。
- **観察ファースト**: `messages.jsonl` / `memory_reasoning.jsonl` のjsonlロギング、`viewer.html` インタラクティブビューア、`generate_video.py` 動画生成と、観察データの取り扱いが体系化されている。提出者の `SUBMISSION_*.md` は「実データに書かれた文字列のみを引用」という規範を守りながら現象を抽出している。
- **拡張機構の実証**: 提出者が `simulation.py` 不変の制約下で日本語プロンプトを後付けした実例（`PROMPT_TEMPLATES` + `sys.argv` 検出）は、本コードベースの拡張容易性を強く実証している。
- **モデル横断方法論**: 多数のモデル（qwen2.5/3, gemma3, llama3.2, sarashina, swallow, elyza, phi4）に対応する config 群と 3-run 観察方法論。

### 改善点・提言
- **乱数シード**: `config.yaml` に `seed` キーを追加し、`random.seed` / `numpy.random.seed` を起動時に注入できると、観察の再現実験が容易になる。現状は同一条件3-runでも発話量が `1390→756→996` と大きく揺れる（`SUBMISSION_ja_d50_qwen25_7b.md` §2）。
- **placeの type 差別化**: 現状 `bar/cafe/library` は文字列ラベルでありtype固有のルールはない。type別に「営業時間」「収容上限の時間変動」「滞在コスト」など世界ルール差を入れると、創発の方向性が大きく変わる可能性。
- **fire_2 の観察可能性**: 提出者の観察で「fire_2 関連語彙が0件」という現象が報告されている。半径8では知覚エージェント数が少なすぎる可能性。fire_1 と fire_2 のスケール差を縮めるか、片方を「目視は近距離だが煙が遠距離まで届く」など知覚モデルを多層化すると、二次火災への対応が観察できるはず。
- **ビジネス/社会科学的なシナリオマッピング**: README に「これは何の現実問題のミニチュアか」（避難訓練設計、SNSデマ拡散、商業施設の混雑制御 等）の対応関係を明示すると B カテゴリの説得力が上がる。
- **テストコードの追加**: `_extract_json_from_text` のネストブレース処理、`get_nearby_agents` の通信ルール、`get_fire_info_for_agent` の Model B 判定など、ロジックが集中しているユニットには単体テストを追加すると後続開発が安全になる。
