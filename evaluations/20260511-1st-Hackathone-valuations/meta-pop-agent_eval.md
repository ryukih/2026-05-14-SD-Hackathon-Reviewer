**評価日**: 2026-05-11

## 概要

仮想売場を舞台に、価格訴求 POP と体験数字 POP（例：糖度 12 度）の 2 種類の販促表示が並ぶ陳列環境をシミュレートするマルチエージェント実装。エージェントは「価格重視」「品質重視」「健康志向」「家族志向」「発見好き」「料理好き」の 6 属性から割り当てられ、入口から売場へ進入する。各エージェントは自身の属性に応じて左側の価格 POP または右側の体験 POP いずれかをターゲットとして選択し、その地点へ移動する。可視化は matplotlib を用いて売場マップ上のエージェント分布として描画される。テーマは「支払数字から体験数字へ」という消費者選好の転換可能性を、属性別の購買意思決定の偏りを通じて検証することにある。

## 評価詳細

### A. 創発設計 — 最終: 1.0/10

#### 評価
本作はエージェントの「選択」を初期化時に完全にハードコードしており、創発の余地が構造的に存在しない。エージェントタイプが `random.choice` で割り当てられた直後に `if agent_type == "価格重視": target = price_pop_area` という二分岐で行動先が確定し、シミュレーションループ中はその目標へ線形補間で近づくだけである。世界ルールも「価格POP座標」「体験POP座標」「入口座標」の3点が固定で置かれるのみで、混雑・知覚境界・情報伝播・エージェント間相互作用といった「世界の制約」は一切定義されていない。世界ルールの設計精度も行動の自由度も最低水準であり、創発ポテンシャルは事実上ゼロ。

#### 根拠
- `main.py:26-31` — `if agent_type == "価格重視": target = price_pop_area / choice = "価格訴求 POP" else: target = experience_pop_area / choice = "体験数字 POP"`（行動が初期化時点で完全決定、ループ内で再評価されない）
- `main.py:62-66` — 移動は `agent["x"] += dx * 0.08` の線形補間のみで、判断・通信・状況認識は存在しない
- `main.py:10-12` — 世界は3座標の定数定義のみで、知覚範囲・コスト・障害物などの世界ルールが無い
- `README.md` — 「📊 実証結果 / 体験数字POPが多数のエージェントに支持される / 価格重視層のみ価格側へ」（"結果"がコードのif分岐で機械的に保証される構造）

### B. 世界設定 — 最終: 4.0/10

#### 評価
「支払数字（価格）から体験数字（糖度12度）へ」というメタPOPのコンセプト自体は小売・マーケティング領域に一定の現実的関連性を持ち、テーマ性は明確である。ただし世界設定としての具体化は浅く、売場は2つのPOPと入口だけ、消費者属性も6種のラベルのみで、価格弾力性・購買確率・属性ごとの効用関数といった社会科学的モデルへの落とし込みが無い。題材は悪くないが、シミュレーションとしての世界の厚みは生まれていない。

#### 根拠
- `README.md` — 「💡 メタPOP構想 / 「支払数字」から「体験数字」へ / 例: 198円 → ❌ / 糖度12度（いちご並み） → ⭕」（コンセプトの問題設定は明示）
- `README.md` — 「エージェント属性: 価格重視 / 品質重視 / 健康志向 / 家族志向 / 発見好き / 料理好き」（属性は列挙されるが、コード上は「価格重視」以外は全て同じ分岐へ流れる）
- `main.py:17-24` — 6種類の属性をランダム付与するのみで、属性ごとの効用差・確率重みなどは未実装
- `main.py:10-11` — 売場が `price_pop_area=[-8,2]` と `experience_pop_area=[8,2]` の2点に縮約されており、世界としての空間構造が極小

### D. 技術実装 — 最終: 0.0/10

#### LLM利用
LLMは一切利用されていない。`openai` / `anthropic` / `langchain` などのSDKインポートも、HTTPによるAPI呼び出しも、プロンプト文字列も、構造化出力パーサも存在しない。READMEは「AIエージェントを仮想売場に配置し、購買行動をシミュレーション」と謳うが、実装上の意思決定は純粋なPythonの `if/else` である。本ハッカソンの主題が "LLM-based multi-agent simulation" である以上、ルーブリックD項目の「0: LLM not meaningfully used」に該当する。

#### メモリ設計
メモリ機構は存在しない。エージェントの状態は `x, y, target, type, choice` のみで、過去観測・他エージェント観測・対話履歴・要約のいずれも保持していない。

#### コード品質・シミュレーション正確性
コードは短く読みやすく、matplotlibでの可視化は機能している。シミュレーションは40ステップを同期的に実行し、競合状態は無いが、`random.seed` が固定されないため再現性に欠ける。LLM未使用かつ意思決定がハードコードであるため、ルーブリックDの主要評価軸（LLM utilization 3pt + Memory 3pt）の合計6点が成立せず、残るコード品質・シミュレーション正確性のみではこの項目の総合スコアを引き上げられない。

#### 根拠
- `main.py:1-2` — `import matplotlib.pyplot as plt / import random`（LLM関連ライブラリのインポートなし）
- `main.py` 全体 — `openai` / `anthropic` / `requests` / `httpx` / プロンプト文字列のいずれも出現しない
- `main.py:33-40` — エージェント状態に `memory` / `history` / `observations` 等のキーが存在しない
- `main.py:16` — `random.seed(...)` の呼び出しが無く再現性が確保されていない
- `README.md` — 「⚙️ 技術構成: Python / matplotlib（可視化）/ エージェントベースシミュレーション」（LLMの記載なし）

## 総評

### 優れている点
- 「支払数字から体験数字へ」というメタPOPのコンセプトは小売・マーケティングの観点で明確な問題提起になっており、テーマ性が分かりやすい
- 87行の小規模スクリプトで、matplotlibによるリアルタイム可視化が動作する完結性
- エージェント属性のラベル設計（価格重視/品質重視/健康志向/家族志向/発見好き/料理好き）にはペルソナ設計の萌芽が見られる

### 改善点・提言
- 本ハッカソンの中核要件であるLLM呼び出しが完全に欠落している。最低限、各エージェントの選択を `if/else` ではなく属性・POP情報・周囲の混雑などを与えたLLMプロンプトに委ねる構成へ置き換える必要がある
- エージェント間の相互作用（他者の選択を観測する、口コミが伝播する等）と、エージェントメモリ（過去の購買履歴・後悔・期待）を導入することで、価格訴求と体験訴求の競合に時間構造が生まれ、創発の余地ができる
- 売場ルール（在庫、滞在時間、価格弾力性、属性別効用関数）を世界ルールとして外部化し、エージェントには「生データのみ」を渡す設計に再構成すると、A. 創発設計のスコアが大きく改善する
- 単一ファイルを `agent.py` / `world.py` / `llm_client.py` / `main.py` 等に分割し、設定はYAML化、`random.seed` を固定して再現性を確保することで、C. 発展性とD. 技術実装の両方が底上げできる
- READMEに将来展望（A/B検証計画、実店舗POS連携、属性比率を変えた感度分析など）を追記すると評価が伸びる

### 一言コメント
コンセプトは魅力的だが、LLM未使用かつエージェントの選択が初期化時のif分岐で機械的に決定される実装であり、「LLMベースのマルチエージェント・シミュレーション」というハッカソンの題意を満たすには至っていない。可視化デモのプロトタイプとして再起動し、LLMによる意思決定・エージェント間通信・メモリの3点を順に導入することを推奨する。
