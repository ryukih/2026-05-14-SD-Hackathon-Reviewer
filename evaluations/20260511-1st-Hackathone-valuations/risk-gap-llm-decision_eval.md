**評価日**: 2026-05-11

## 概要

登山におけるリスクの客観値と主観認識のギャップ（Risk Gap）、および「止まれない意思決定の力学」を扱う FastAPI ベースの Web デモ。シナリオは雪山登山隊で、リーダー 1 名と複数のパーティーメンバーが「行動する（trek）」か「停止する（rest）」かの二択を毎ステップ判断する。世界側は天候・体力・距離・時間などの数値変数を `risk_simulator.py`（実体は `app.py` 内）で純粋にシミュレートし、客観リスク値を生成する。LLM（ホストの Ollama、デフォルト llama3.2）はそのリスク数値を受け取り、`GUIDE_PERSONAS` 辞書で定義された 4 種のガイドペルソナ（楽観的／慎重派など）に応じて、リーダーのフォールバック発言とパーティーメンバーのセリフを生成する。ユーザーはブラウザ UI 上で `/api/advance`（1 ステップ進行）と `/api/decide`（continue / llm_stop）を操作し、実在の山岳事故 3 件をベースに「客観的にはリスクが高くても登り続けてしまう心理」を観察する。テーマは、リスク認識のバイアスとサンクコスト・ピアプレッシャーが意思決定にどう作用するか、また LLM ペルソナの違いが助言内容と最終判断にどう影響するかを実験的に検証することにある。

## 評価詳細

### A. 創発設計 — 最終: 3.0/10

#### 評価
世界ルールの設計精度は高い（R_obj/R_subj/Gap/Cost_stop が線形式と閾値で厳密に定義され、ペルソナ別に初期値プリセットが用意されている）一方、**エージェントの行動自由度はきわめて低い**。LLM の役割は設計書で明示的に「判断主体ではない」と定義されており、選択肢は `trek`/`rest` の二択＋テキスト生成に限定される。さらに、これは **単一意思決定者** のシミュレーションであり、複数エージェントの相互作用による創発の余地が構造的に存在しない。パーティチャット（引率＋隊員2名）も 1 回の LLM 呼び出しで3人分のセリフを一括生成する単一プロンプト構成で、エージェントとしての独立性・自律性を持たない。テンプレ・フォールバック（`_party_member_lines` 等）が主体のため、設計レベルでの創発ポテンシャルは限定的。

#### 根拠
- `apps/demo_web/risk_simulator.py:103-141` — `metrics()` で R_obj/R_subj/Gap を厳密な線形式 `w1*env_avg + w2*human_avg + w3*time_pressure` で算出（世界ルールは精密）。
- `apps/demo_web/app.py:616-650` — `_ollama_guide_plan_next_step` は LLM に `"trek"` か `"rest"` の二択のみを問う（行動空間が極小）。
- `apps/demo_web/app.py:345-498` — `_leader_fallback_line` / `_party_member_lines` は条件分岐＋固定プールから乱択するテンプレ。
- `docs/presentation/v2.md` — "判断はルール、納得はLLMが担う" — LLM を判断主体から明示的に除外する設計思想。
- `docs/design/high_level_design_v2.md` — "LLMは：その歪みを露出させる装置"（LLM の役割を補正・説明に固定）。

### B. 世界設定 — 最終: 7.0/10

#### 評価
題材は **登山などレジャー行動における「客観リスクと主観認識のギャップ（過信）」と「止まれない意思決定の力学」** で、軽井沢スキーバス・知床遊覧船・那須雪崩など実在の重大事故を抽象化する社会科学的・安全工学的な視点を持つ。R_obj（環境・人・時間の合成）と R_subj＝R_obj−バイアス、Gap、Cost_stop（引き返し困難さ）という構成は、心理学・経営判断（サンクコスト・社会的圧力）の知見を素朴に・しかし筋よくモデル化しており、業務的（安全管理・運行判断）にも関連性が高い。一方で「世界」自体は基本的に単一行程（登山）の数値遷移であり、空間的・社会的構造の豊かさは限定的。原性は中〜高、業務的価値は中〜高。

#### 根拠
- `docs/design/high_level_design_v2.md` — "軽井沢スキーバス転落事故、知床遊覧船沈没事故、那須雪崩事故" を背景に挙げ "合理的に見える意思決定の連続が破綻する現象" を主題化。

### C. 発展性 — 最終: 6.5/10

#### コード拡張性
FastAPI + 数値シミュレータ + 静的フロントの分離が明確。`RiskSimulator` が dataclass で表現され `new_simulator(**overrides)` でパラメータ上書きが可能。ペルソナ設定は `GUIDE_PERSONAS` 辞書に外出しされており、新規ペルソナの追加は容易。Docker / docker-compose 完備、ログ出力先も `DEMO_OUTPUT_DIR` で切替可能。判断トリガを `JUDGMENT_*` 定数で抽象化しており、新たな判断条件を追加する余地がある。一方、`app.py` がやや肥大化（fallback テンプレ・LLM・API・チャット整形が同居）しており、LLM クライアントや会話生成は別モジュールに分けると拡張しやすい。

#### 将来展望
プレゼン v2 で「振り返りの自動生成 / 説明のパーソナライズ / 判断比較の提示」の 3 点を明示し、設計書 vNext（`rp_ui_and_simulation_vnext.md`）で多段判断・シード付き RNG・構造化出力など具体的な技術ロードマップを提示。デモ実装ガイドにも「DB を後から SQLite に移せる」など段階的拡張への配慮がある。ただし、いずれの方向も既存テーマの深化に留まり、本質的に新しい研究・業務的拡張（他ドメイン、他者との相互作用、確率分布化、A/B テスト機能など）は限定的。

#### 根拠
- `apps/demo_web/risk_simulator.py:408-416` — `new_simulator(max_steps=..., **overrides)` で柔軟な初期化を提供。
- `apps/demo_web/app.py:36-109` — `GUIDE_PERSONAS` をデータ駆動で定義しペルソナ追加が容易。
- `docker/docker-compose.yml` 配置と `docs/CLAUDE.md` の Docker 運用文書（拡張性のあるデプロイ基盤）。
- `docs/presentation/v2.md` — "振り返りの自動生成 / 説明のパーソナライズ / 判断比較の提示" を将来拡張として明記。

### D. 技術実装 — 最終: 7.0/10

#### LLM利用
Ollama HTTP API への `requests` ベースの呼び出しを `_ollama_generate` に集約し、`temperature` / `num_predict` をユースケース別に切り替え（説明文 0.3、会話 0.7、判断 0.35）。JSON 抽出に堅牢な複数戦略（`{...}` 抜き出し→正規表現フォールバック）を実装。プロンプトでは「数値は捏造禁止」「JSON のみ」を明示し、応答失敗時は決定論的フォールバック（6 ステップごとの休憩、テンプレチャット）に退行する設計で、UX が壊れない。一方、`except Exception: pass` で例外を握りつぶしており、運用時のログが弱い。

#### メモリ設計
`guide_chat` を最大 120 行で保持し、直近 N 行をプロンプトに「重複回避用」「文脈用」として渡す。`judgment_events` で意思決定履歴を構造化して保持。ただし、メモリは事実上「直近会話のログ」であり、要約・抽出・忘却などの能動的メモリ機構はない（vNext で要約フィールドの追加が言及されているが未実装）。エージェント固有の記憶や信念モデルも持たない。

#### コード品質・シミュレーション正確性
`from __future__ import annotations`、type hints、dataclass、docstring が一貫している。命名は日本語コメント＋英語識別子で読みやすい。`_clamp01`、`_scale`（max_steps に応じてステップ毎悪化量をスケール）など細部のチューニングが丁寧。`rng_seed` でシード固定し、`metrics()` を純関数化してスナップショットの再現性を担保。`legacy_decision` フラグで旧仕様との後方互換も持たせている。FastAPI の状態はモジュールグローバル変数（`_sim`、`_guide_agent_ui`）で、マルチプロセス・マルチユーザでは race condition の余地があるが、ローカルデモ用途では許容範囲。

#### 根拠
- `apps/demo_web/app.py:226-247` — `_ollama_generate` で Ollama API を一元化、`temperature`/`num_predict` をパラメータ化。
- `apps/demo_web/app.py:250-285` — `_parse_guide_json` の多段 JSON 抽出フォールバック。
- `apps/demo_web/app.py:241-247` — `except Exception: return None` で例外を握りつぶし（観測性は弱い）。
- `apps/demo_web/app.py:586-611` — プロンプトに「数値は捏造しない」「直近会話（重複回避用）」を明示。
- `apps/demo_web/risk_simulator.py:65-91` — `rng_seed` と dataclass による再現性、`append_guide_chat` の上限制御。
- `apps/demo_web/risk_simulator.py:93-141` — `metrics()` が副作用なしの純関数。
- `apps/demo_web/risk_simulator.py:165-166` — グローバル `_sim` 変数（並行性に対する弱さ）。

## 総評

### 優れている点
- 実在事故を背景にした「Risk Gap × 止まれない意思決定」というテーマ設定が骨太で、業務的・社会的価値が明確（B 軸の強み）。
- 数値モデル（R_obj/R_subj/Gap/Cost_stop）を Python に固定し、LLM を「説明・補正レイヤ」に限定する役割分離が一貫しており、ハルシネーションで指標が崩れない設計。
- ペルソナプリセット・判断トリガ・判断履歴・seed 制御など、デモ／検証用の機能が丁寧に整備されている。
- FastAPI + Docker + 静的フロント + 詳細な設計ドキュメント群（高レベル設計・RP 設計・vNext ロードマップ）で運用・引き継ぎがしやすい。

### 改善点・提言
- **創発性の弱さ**: 単一意思決定者シミュレーションのため、複数エージェントが独立に観測・判断・通信する構造（例：隊員それぞれが独自の `metrics` 観測ノイズと記憶を持ち、引率に進言する／互いに相互作用する）に拡張すると、創発ポテンシャルが大きく伸びる。
- **LLM の行動空間が極小**（trek/rest の二択）: アクション空間を拡張するか、隊員に独立した内省・記憶・発話戦略を持たせれば、本来の「マルチエージェント LLM シミュ」の評価軸で評価値が上がる。
- **メモリ機構の深化**: 現状は直近チャットの貯蔵に留まる。要約・信念・経験エピソードなどの能動メモリを導入すれば、ペルソナの一貫性と長期挙動が強まる。
- **例外処理の観測性**: `except Exception: pass` を `logging.exception` 等に置き換え、Ollama 失敗の頻度・要因を可視化すべき。
- **モジュール分割**: `app.py` の会話テンプレ・LLM クライアント・チャット整形を別ファイルへ分離するとテスト容易性が上がる。

### 一言コメント
「LLM は判断装置ではなく説明装置」という割り切りが明確で、社会的に意義あるテーマを最小モデルで具体化した堅実な作品。一方で **マルチエージェント LLM シミュレーション** という評価軸では、エージェントの複数性・自律性・相互作用が薄く、創発設計の評価は伸び悩む。
