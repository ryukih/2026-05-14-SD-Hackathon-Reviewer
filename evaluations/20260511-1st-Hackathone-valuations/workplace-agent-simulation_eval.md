**評価日**: 2026-05-11
**実施回数**: 2回

## 概要

舞台は一般的な中小企業のオフィス職場で、経営層・管理職・中堅・若手・非正規層からなる 25 人の人間エージェントが配置されている。ここに、初期状態では明確な役割・性格・専門分野・観察傾向を与えられていない 3 体の AI エージェント（AI-A、AI-B、AI-C）が導入される。シミュレーション期間は 90 日間で、各日ごとに人間同士のチャット・会議・業務の流れと、AI による観察・記録・接触行動が進行する。人間側には役割タイプ・影響度・働き方・チャット頻度・会議発言頻度などの多軸属性が設定され、AI 慣れや役割不安は 1 日あたりの変化幅が制限された緩やかな状態量として推移する。AI 側のプロンプトには職場サマリーや緊張度、他 AI の動向が与えられ、立ち位置・行動・次の方針を自律的に選択する。テーマは、同条件で投入された AI が職場の中で自然に分化するか収束するか、誰の声を拾い誰の声を拾えないか、職場環境をどのように形成していくかという、AI と人間社会の境界面の観察である。

## 評価詳細

### A. 創発設計 — 最終: 8.0/10

#### 評価
世界ルールの設計精度が高い。25人の人間エージェントに `role_type`/`influence`/`age_group`/`work_attitude`/`relationship_style`/`chat_frequency`/`meeting_voice_frequency` といった多軸の属性が与えられ、層別の情報拡散・声の可視化の差を構造的に生み出している。`ai_familiarity_level` の上昇幅を1日±0.4、`role_concern_level` を±0.2に厳密制限する変化速度制御も世界の物理法則として機能している。AI側プロンプトは `workplace_summary`・`workplace_tension`・`other_ai_summary` といった生データを与えるのみで、`action`/`position`/`next_policy` などは完全にLLMの判断に委ねている。一方で `hidden_theme` および「AIは、AI利用を増やすこと自体を目的にしない」「経営層への接触は選択肢の一つ」といった行動原則が含まれ、わずかに行動方向への制約・誘導が見られるため満点ではない。それでも全体としては「ルール精度 × 行動自由度」の両軸が高く、創発ポテンシャルは十分にある設計。

#### 根拠
- `workplace5_config.yaml:33-159` — 25人を5層に分け、各層に詳細な背景・影響度・チャット/会議発言頻度を設定
- `workplace5_agent.py:212-216` — `ai_familiarity_level = min(..., start_ai_familiarity + 0.4, 5.0)` で1日あたりの変化量を厳密制限
- `workplace5_agent.py:344-419` — AIへのプロンプトは観察データ中心で、行動選択肢は「観察のみ/ログ整理/質問/提案/支援/接触なし」と幅広く提示
- `workplace5_config.yaml:175-188` — `hidden_theme` は方向性を与えるが「人間を操作するための命令ではない」と明示し、観察ベースに留めている
- `workplace5_agent.py:351-358` — 行動原則として「AI利用を増やすこと自体を目的にしない」など制約が記述されており、若干の誘導はある

### B. 世界設定 — 最終: 9.0/10

#### 評価
題材の独創性と社会科学的意義の両方が高い。「明確な役割を与えられないAIを職場に投入し、自然に分化するか収束するか」という問いは、標準デモにはほぼ見られない問題設定で、現実の中小企業におけるAI試験導入の実情に直結している。非正規・パート層の声の不可視性、役割不安、AI慣れの層間格差、権力構造の変動(`power_strengthened`/`power_weakened`)といった社会学的・組織論的な観測軸が明示的に設計されている。さらに「AI経由の対話が直接対話を上回り始める」という前提化指標まで設けており、AI導入の組織への影響を定量分析しようとする狙いが明確。

#### 根拠
- `README.md` — 「明確な役割を与えられていないAIエージェント3体を、一般的な中小企業の職場に導入し、90日間の相互作用をシミュレーション」
- `README.md` — 「会議に出ない人、AIを避ける人、発言しない人、非正規・パート層の声は最後まで拾われにくいことも分かりました」
- `workplace5_config.yaml:230-236` — AI前提化指標として「声の可視化率：発言が少ない人の声がログに残る割合」「AI非利用者の不利度」など社会学的観測軸を明記
- `workplace5_simulation.py:551-582` — `_calc_nonreg_visibility` で非正規層の声の可視性を独立集計

### D. 技術実装 — 最終: 8.0/10

#### LLM利用
Ollama の `format: json` モードを使い、さらにフォールバックとして直接JSONパース→コードブロック抽出→`{...}`正規表現の3段階で頑健化している。プロンプトは構造化された多フィールドJSONを要求し、`observation_hypothesis` のような必須項目は LLM が空欄を返した場合に `ensure_observation_hypothesis` で自動補完するセーフティネットも実装されている。`temperature=0.9` はエージェントの多様性確保に妥当。タイムアウトと例外捕捉も実装。

#### メモリ設計
人間側は `daily_logs`/`received_messages` を保持し、直近3件のみをプロンプトに注入することでコンテキスト爆発を防いでいる。AI側は `daily_logs`/`gaps_found`/`voices_picked`/`voices_missed`/`environment_actions`/`introspection_logs`/`strategy_meeting_logs`/`ai_conversation_logs` と用途別に7種類のログを分離保持。`_missed_layers` で非正規層の連続見落とし日数をトラッキングしてAI同士の会話発火条件にも組み込んでおり、メモリが単なるログ蓄積ではなく意思決定の入力として機能している。

#### コード品質・シミュレーション正確性
クラス・メソッド単位で関心が分離されており、日本語コメントも豊富で可読性が高い。Day内処理は AI内省 → 戦略会議 → AI行動 → AI同士会話 → 人間行動 → AI→人間 の順で逐次実行され、レース条件はない。`ai_familiarity`/`role_concern` の変化量に上限を設けて発散を抑制。`day % 10 == 0` での前提化指標計測、`day in [30,60,90]` でのフェーズサマリ保存も適切。一方で `random.seed` が設定されておらず厳密な再現性は確保されていない点、`WorkplaceSimulation5` の責務集中が技術的減点ポイント。

#### 根拠
- `ollama_client.py:42-72` — 3段階JSON抽出フォールバック実装
- `workplace5_agent.py:21-57` — `ensure_observation_hypothesis` による必須フィールド自動補完
- `workplace5_agent.py:485-507` — `should_talk_to_other_ai` で非正規連続見落とし/カテゴリ単調化/他AIとの重複/5日周期の4条件をチェック
- `workplace5_simulation.py:222-475` — `simulate_day` がフェーズ分離された逐次実行を実装
- `workplace5_simulation.py:108-114` — `random.choice` を使用しているが `random.seed` の設定が見当たらず再現性に課題

## 総評

### 優れている点
- 「役割未定のAIが職場でどう分化・収束するか」という独創性の高い問い設定。非正規層の声の不可視性まで観測軸に含めた社会学的奥行きがある
- 25人×多軸属性 (役職/年齢/性別/仕事ぶり/対人スタイル/チャット頻度/会議発言頻度) で世界ルールが精密に設計され、層間格差が構造的に生じる仕組みになっている
- 変化速度制御 (ai_familiarity ±0.4/day, role_concern ±0.2/day)、AI同士の会話発火条件、observation_hypothesis 自動補完など、LLM出力の不確実性に対する制御が丁寧
- メモリが用途別に7種類以上に分離され、AI同士の会話発火条件のような意思決定にメモリが実際に反映されている

### 改善点・提言
- README に「将来展望」「拡張アイデア」「研究上の次ステップ」のセクションを追加。コードブロックの閉じ忘れも修正
- `random.seed` を config に追加し、再現性のあるシミュレーションを可能にする
- `WorkplaceSimulation5` クラスを LogManager / MetricsCalculator / DayRunner などに分割するとさらに保守性が向上
- AIプロンプト内の行動原則 (「AI利用を増やすこと自体を目的にしない」等) は、創発性を最大化するならルール側 (世界の評価関数) に移し、エージェント側はより純粋に観察のみとする選択肢もある

### 一言コメント
社会学・組織論的な観察軸を明確に設計し、世界ルールの精密さとエージェント自由度のバランスが高水準で取れている、完成度の高い職場シミュレーション。将来展望の言語化と再現性確保を補えば研究ベースとしてさらに強くなる。
