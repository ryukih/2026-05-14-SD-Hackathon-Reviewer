## 概要

火星探査ミッションを舞台にした LLM マルチエージェント・シミュレーションで、チーム構成に「犬（ワンコ、Pochi）」を入れた場合と「化学者（Joe）」を入れた場合のエージェントの行動・ストレス・探査効率を比較する研究プロジェクト。各条件のクルーは 9 名で、火災イベントやエリア制約などの環境ストレッサーが世界側に組み込まれている。エージェントは GPT-4o-mini で駆動され、ペルソナごとの役割と性格に基づいて行動・対話・記憶を生成する。実験は Dog 条件 vs Chemist 条件で各 10 回（探索 3 回→検証 10 回）実施され、ストレス指標（d=0.57 で Dog チームが低い傾向）、固着パターンの普遍性、小サンプルでの効果量過大推定（d=1.41→0.57）などの分析結果が `Analysis/` 配下の PDF にまとめられている。テーマは「非人間のチームメンバー（犬）が、長期閉鎖環境下のクルーの心理・協調にどのような効果を持つか」という、極限環境心理学および LLM エージェントの行動固着特性の検証である。

### A. 創発設計 — 最終: 5.0/10

#### 評価
世界ルールの設計精度は高く、複数のplace（habitat/lab/site_A-D 等）に座標・capacity・occupancy_rate、火災（位置・強度・半径・距離）、ストレス、コミュニケーション半径・同一エリア制約などが定量的に与えられている。一方で、行動の自由度は明確に制限されている。`agent.py` のミッションプロンプト内に「ストレスレベルが0.7以上の場合は…habitatに戻って休息」「ストレス0.9以上は外部探査を継続しないでください」と直接的な行動指示が書かれており、各役職ごとの責任（パイロットは安全帰還、生命科学者は Site_D を重視 等）まで指示している。debrief には「ワンコの状況も毎日報告することが重要」とまで書かれている。このため、世界ルール精度=高、行動自由度=中〜低 のアンバランスとなり、両軸の調和では中位に収束する。

### D. 技術実装 — 最終: 6.0/10

#### LLM利用
GPT-4o-mini を Chat Completions で呼び出し。プロンプトは「メッセージ決定（位置情報なし）」→「メッセージ送信」→「行動決定（位置情報+メッセージ込み）」の2段階に分かれており、情報の段階的開示が設計されているのが秀逸。JSON抽出は中括弧深度マッチによる堅牢な実装（`_extract_json_from_text`）。フォールバックパス（テキスト解析→stay）あり。一方で、temperature=0.7 のみで再試行・スキーマ検証なし、`max_tokens=200` と短く、JSON が途中で切れる可能性がある。

#### メモリ設計
LLM が自分で次に覚える内容を `memory` フィールドに書き、それを次ステップのプロンプトに直近 `memory_size` 件入れる「自己フィードバック型」メモリ。`memory_limit` で件数も管理。message も `message_history_limit` / `message_context_size` で別管理。単なるログダンプではなく、エージェントが自律的に要約する仕組みになっており、設計としては良好。

#### コード品質・シミュレーション正確性
2段階シミュレーション（全員メッセージ→全員行動）はシンクロナスに進行し、レース条件は無さそう。観測ログ（agent_logs / memory_reasoning.jsonl / messages.jsonl）も整備されている。ただし以下の品質課題あり：
- `agent.py:349-350` に `print("DEBRIEF LENGTH:", ...)` のデバッグ出力が残存
- `simulation.py:436-437` で `outside_position = (0, -self.half_space_size)` が2行重複
- `ollama_client.py:46-61` に `return` 以後の到達不能コード（旧 `responses.create` 呼び出しが残存）
- `OllamaClient` という名前のまま OpenAI を呼ぶ命名の矛盾
- `simulation.py:546-578` に同じCSV保存処理が2回書かれている（`run` メソッド内の重複）
- 乱数シードが指定されていないため再現性が低い（`random.choice`, `random.randint` を使用）
- `gender = random.choice(["male", "female"])` で犬にも性別が割り当てられるなど細部の整合性

#### 根拠
- `ollama_client.py:35-44` — Chat Completions 呼び出し、例外時は空文字返却
- `ollama_client.py:46-61` — `return` 後の到達不能コード
- `agent.py:563-598` — `_extract_json_from_text` の堅牢な中括弧マッチング
- `agent.py:713-722` — LLM の `memory` 出力をメモリ列に追記し件数制限
- `agent.py:171-188` — メモリ・メッセージのコンテキスト抽出ロジック
- `simulation.py:342-466` — 2フェーズ（メッセージ→行動）の同期ステップ実装
- `simulation.py:436-437` — `outside_position` 重複代入
- `agent.py:349-350` — `print` デバッグ残存
- `simulation.py:111-127` — 統計取得構造（places ごとのoccupancy/agents_in_place）

## 総評

### 優れている点
- 「火星アナログ環境において犬（コンパニオンアニマル）がクルーのストレスや行動に与える影響を比較検証する」という、ハッカソンとして十分にオリジナルかつ社会科学的・ミッション運用的な切り口
- 2段階のプロンプト（位置情報を伏せたメッセージ決定 → メッセージ込みの行動決定）による情報非対称性の設計が丁寧
- LLM 自身に「次ステップで覚えること（memory）」を生成させる自己フィードバック型メモリが、単なるログダンプではなく構造的に意味を持つ
- 火災情報は半径内のエージェントにのみ数値データで提示するなど、知覚境界が定量的にモデル化されている
- 検証3回→本実験10回、効果量 d=0.57（小サンプルでは d=1.41 と過大推定）という統計的に堅実な分析姿勢

### 改善点・提言
- ミッションプロンプトに「ストレス0.7以上ならhabitatへ戻れ」「Pilot は帰還を考慮」「Geologist は Site_B を重視」など具体的な行動指示や役職別の優先度が大量に書かれており、行動の自由度を下げて創発を妨げている。世界ルール（ストレスが何点でどう増減するか、未探査だと何が失敗とみなされるか）だけを与え、戦略は LLM に任せる方向に整理することで創発設計のスコアは大幅に向上する
- `update_stress` などのドメインロジックがplace名ハードコードで、`OllamaClient` 命名・到達不能コード・デバッグprint・CSV保存処理の二重化など、コード衛生面の整理が必要
- 役職・クルー名・ミッション概要が `agent.py` に長文で埋め込まれているので、`config.yaml` または別ファイル（debrief 同様）に切り出してホットスワップ可能にすると拡張性が上がる
- 乱数シードを config に追加して再現性を確保する
- README 自体には将来展望や技術的拡張案がほとんど書かれていないため、PDF に頼らず本文中にも要約するとよい

### 一言コメント
「ペット犬の有無」というニッチで魅力的な仮説と統計検証まで踏み込んだ意欲作で、世界ルールの設計はしっかりしている。一方でプロンプトに行動指示が大量に書き込まれているため、創発というよりは「LLMに丁寧にロールプレイさせる」シミュレーションになっており、世界設定・解析の強さと比べて創発設計の評価は伸び切らなかった。
